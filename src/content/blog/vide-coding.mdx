# Claude.md 编写规则

## 渐进式信息

* 将特定于任务的指令保存在项目某处具有自描述名称的单独 Markdown 文件

```
agent_docs/
  |- building_the_project.md
  |- running_tests.md 
  |- code_conventions.md
  |- service_architecture.md
  |- database_schema.md
  |- service_communication_patterns.md
```

在 CLAUDE.md 文件中，你可以包含这些文件的列表以及对每个文件的简要描述，并指示 Claude 决定哪些文件（如果有）是相关的，并在开始工作之前阅读它们。或者，要求 Claude 先向你展示它想阅读的文件以供批准，然后再阅读。


---

CLAUDE.md 文件可以使用 @path/to/import 语法导入其他文件：


```

# Additional Instructions
- Git workflow: @docs/git-instructions.md
- Personal overrides: @~/.claude/my-project-instructions.md
```

---

我建议不要写太长，我一般会包括：

常用命令（claude会用来执行）
目录结构（claude会去对应位置找相关代码）
项目规范链接（claude也会去遵循，一些很关键你要它遵守的，可以在claude.md里强调一下）
一般的工作流程（比如，先写哪部分再写哪部分，最后写测试用例等，不过我会要求不进行测试验证，我会交给gemini去测试和debug）

# 怎么做设计

首先你得知道你想要什么样的设计 -> 将这份审美变成 design system  -> 最后利用这个desgin system 来vide coding

**如何得到  design system**
 
1. 找到你喜欢的产品，你觉得优秀的设计
2. 把你觉得好的地方描述出来
- 它的颜色怎么用？主色是不是很克制？背景是偏灰还是偏白？
- 它的间距、元素密度：是很紧凑的 IDE 风，还是很宽松的 Notion 风？
- 它的层级：标题、正文、次要信息之间的字号差距有多大？
- 它给你的感觉：冷静、玩具感、工业感、学术感、未来感…？

这一步的目标：把别人 UI 的“感觉”拆成文字标签

3. 拆成几块“我将来要控制的参数”

问自己一个问题——以后我让 AI 画 UI，要固定哪些维度？

通常可以先从这几块入手：

- 布局 & 空间（最大宽度、左右留白、常用间距节奏）

- 圆角 & 阴影（圆角大小、什么时候用阴影/边框）

- 字体 & 层级（标题/正文/辅助文字的字号、粗细）

- 颜色角色（背景、主色、危险色、成功色、文本颜色）

- 组件习惯（按钮长什么样、输入框长什么样、卡片长什么样）

- 氛围/情绪（冷静、严肃、可爱的程度、信息密度）

你不用一开始就写得很精确，可以先写「偏好」，比如：

- “我喜欢偏宽松一点的布局，不要像 IDE 那么密”

- “圆角不要特别夸张的 999px，除非是主 CTA 要有一点特色”

- “颜色尽量灰度化，主色只用在操作和高亮上”

这一步是帮自己确定：以后这几个维度，都是我要给 AI 讲清楚的，不允许它乱来。

4. 选出“第一版默认值”，哪怕只是模糊的 ￼

这时你会发现一个问题：

“我知道方向，比如喜欢圆角偏小、间距偏宽，但具体多大、多宽？我也不确定。”

没关系，你可以先随便选一组“看着顺眼的数字”，当第一版 token。比如：

- 间距：4 / 8 / 12 / 16 / 24 / 32（你太熟了）

- 圆角：6 / 12（默认用 6，重要卡片用 12，如果哪天觉得丑再改）

- 字号：12 / 14 / 16 / 20 / 24 / 28（正文 14 or 15，标题 H2 20，H1 24–28）

- 颜色：

 ▫ 背景：#050814（深）或 #F3F4F6（浅），可以先选一个模式

 ▫ 主色：你眼缘最好的一个蓝 / 紫，例如 #4F46E5

 ▫ 危险色：先用经典红 #EF4444

 ▫ 成功色：先用 #22C55E

 ▫ 文本色：深背景用 #E5E7EB，浅背景用 #111827 / #374151

这一步的思想是：不要求“完美设计师”，先有一份“可迭代的默认值”。

你以后在真实项目里用了一阵子，觉得“这个圆角看久了不爽”，再一口气改全局 token。

5. 用“句子”把这些默认值写成

现在你已经有：

- 看过的参考 UI 的感受

- 想要控制的维度

- 初步的数字 / 颜色默认值

接下来就可以写那份你刚才说“我不知道怎么得到”的设计系统了。

我给你一个极简骨架，你可以照着填：

```
# Luma UI Vibes v0.1

[Product]
- 名称：Luma
- 使命：精确点亮视频里的知识点，让自学者有“被好老师带着走”的感觉
- 用户：国内大学生 & 自学者，习惯用 B 站/YouTube 学东西
- 气质关键词：冷静、专业、干净、稍微有一点未来感，但绝不花哨

[布局 & 空间]
- 页面最大宽度：1200px，内容居中
- 水平左右留白偏宽，避免贴边
- 间距使用刻度：4 / 8 / 12 / 16 / 24 / 32
- 默认模块间距：24（上下）、16（内部元素）

[圆角 & 阴影]
- 默认圆角：6
- 重要卡片、模态：12
- 原则：边框和阴影不要同时很重，优先细边框 + 很轻的阴影，仅在需要“浮起”的地方用

[字体 & 层级]
- 字体：系统 UI 中文 + 英文无衬线
- 正文字体大小：14–16，行高约 1.6，易读优先
- 层级示例：
  - H1：28，半粗
  - H2：22，半粗
  - H3：18，中粗
  - Body：14/15，常规
  - Caption：12
- 原则：层级差距通过字号 + 粗细 + 间距共同体现，避免用颜色来做层级

[颜色角色（深色模式示例）]
- 背景（page）：#050814 深色渐变（或近似色）
- 背景（surface）：#0B1020，用于主要卡片、面板
- 分割线/边框：#1F2933 附近的深灰
- 主色（primary）：#4F46E5，只用于主要操作、高亮知识点
- 成功（success）：#22C55E，用在“掌握良好”的状态
- 危险（danger）：#EF4444，只在真正错误/危险操作时使用
- 文本（主）：#E5E7EB
- 文本（次）：#9CA3AF

[组件习惯]
- Button：
  - 主按钮：背景主色 + 白字，圆角 999px，突出 CTA
  - 次按钮：边框按钮（outline），圆角 6，颜色更克制
  - 禁用态：降低对比度，不用鲜艳颜色
- Input：
  - 圆角 8，1px 边框，默认浅灰边框
  - Focus：边框主色 + 稍微发光的阴影，而不是整块底色变主色
- Card：
  - 圆角 12，轻边框或轻阴影二选一
  - 内部有清晰的标题 + 辅助信息 + 主操作区

[氛围 / 文案]
- 整体感觉：像一个耐心的教练，而不是严厉的老师
- 对“错误/未掌握”的反馈：多用“需要巩固”、“不太确定”，少用“错误”、“失败”
- 对话语气：简洁、直白，但不过于冷冰冰，有一点“帮你一起解决”的语气
```

6. 在真实 UI 里“踩坑 → 修订”，让设计系统长出来

可以做一个小迭代循环：

1. 把上面这份 ‎`Luma UI Vibes v0.1` 贴进你的 AI（Claude / Gemini / Stitch）

2. 让它按这个指南，帮你改 Luma 的某一个具体页面（比如“视频学习主界面”）

3. 你盯着看：哪些地方“跟你脑子里的 Luma 不一样”？

 ▫ 觉得背景太黑/太亮？改颜色 token

 ▫ 字号太小/太大？改字号 token

 ▫ 按钮太显眼/不够突出？改按钮规则

4. 每次这样调一调，就把修改同步回 ‎`UI_VIBES.md`，版本号从 v0.1 → v0.2 → v0.3…

最后你会发现：那份设计系统不是你坐在那“想出来”的，而是踩着项目往前走，一点一点“长”出来的。
